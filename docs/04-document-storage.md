# Module 4 — Document Storage & E-signing

## Overview

All documents (contracts, service agreements, payslips) are stored as PDFs in Vercel Blob. E-signing uses the simplest legally valid approach for the Netherlands.

---

## E-signing Approach: Simple Electronic Signature (SES)

Dutch law (eIDAS Regulation, implemented in Dutch Civil Code §3:15a) allows electronic signatures for uitzendovereenkomsten. We use a **Simple Electronic Signature (SES)** — the lowest tier, but fully sufficient for these contracts.

### What counts as a valid SES in NL:
- Student receives a unique link to review the contract PDF
- Student reads the contract
- Student checks a box: *"I agree to this contract and confirm my identity"*
- System records:
  - Timestamp (UTC)
  - IP address
  - User agent (browser/device)
  - User ID (authenticated session)
- Confirmation email sent to student

This approach is:
- ✅ Legally valid for uitzendovereenkomst in the Netherlands
- ✅ Free (no third-party signing service)
- ✅ Simple to implement
- ✅ Auditable

For higher-risk documents (e.g., future Fase C permanent contracts), consider upgrading to AES via SignRequest or DocuSign.

## Signing Flow

```
1. Admin generates contract (PDF created, stored in Vercel Blob)
2. Admin clicks "Send for signing"
3. Student receives email with a signing link (unique, time-limited token)
4. Student opens link → sees contract PDF inline
5. Student must scroll to bottom before checkbox becomes active (optional UX guard)
6. Student checks confirmation box + clicks "Sign"
7. System records: timestamp, IP, user agent, userId
8. PDF is marked as signed (metadata appended or stored separately)
9. Admin receives confirmation email
10. Contract status → SIGNED → ACTIVE
```

### Signing Token
- One-time use JWT with 7-day expiry
- Tied to the specific contract ID + student user ID
- Invalidated after use

## Vercel Blob Structure

```
/documents
  /contracts
    /{contractId}/contract-unsigned.pdf
    /{contractId}/contract-signed.pdf      ← after signature
  /agreements
    /{employerId}/service-agreement.pdf
  /payslips
    /{studentId}/{year}-{month}.pdf
  /uploads
    /{studentId}/id-document.{ext}
```

## PDF Generation

Use **`puppeteer`** with `@sparticuz/chromium` (Vercel-compatible headless Chrome):

1. Contract template stored as an HTML template (with Handlebars/template literals)
2. At generation: merge data into HTML
3. Render HTML → PDF via headless Chromium
4. Upload PDF to Vercel Blob
5. Return blob URL → stored on Contract record

Alternative: `pdf-lib` for simpler documents (no complex layout), but puppeteer gives better results for contract-style documents.

## Security

- Vercel Blob URLs are private by default (use `put` with `access: 'private'`)
- Signed URLs generated on demand (expire in 1 hour) — never expose permanent blob URLs
- Only `ADMIN` can access all documents
- Students can only access their own documents
- Employers can only access their service agreement + (future) their invoices
- BSN number is **never** printed in full on any PDF — only last 4 digits on the contract document

---

## Access Control for Documents

```
GET /api/documents/:id → check session role:
  ADMIN    → always allowed
  STUDENT  → allowed if document.studentId === session.user.studentId
  EMPLOYER → allowed if document.employerId === session.user.employerId
  else     → 403
```
